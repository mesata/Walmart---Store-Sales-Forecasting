##
ყველა გაშვება (run) ილოგება MLflow-ში DagsHub-ის პლატფორმაზე. მეტრიკა: WMAE (Weighted Mean Absolute Error — შეწონილი საშუალო აბსოლუტური შეცდომა), რომელიც შეჯიბრის ოფიციალური მეტრიკაა. მასში სადღესასწაულო კვირებს 5-ჯერ მეტი წონა აქვთ, ვიდრე ჩვეულებრივ კვირებს.



$$WMAE = \frac{\sum weight_i \cdot |y_i - \hat{y}_i|}{\sum weight_i}$$

```
WMAE = sum(weight_i * |y_i - yhat_i|) / sum(weight_i)
weight_i = 5 if week_i is a holiday week else 1
```

## XGBoost
ამ მოდელის გაწვრთნისას თავიდან დავიწყე base feature engineering-ით (`WalmartTransformer` კლასი), რომელიც მოიცავდა მარტივი ფიჩერების შექმნას: `Year`, `Month`, `Day`. ეს გავაკეთე, რადგან ერთი დროის ფიჩერი დამეშალა და მოდელისთვის მეტი საშუალება მიმეცა დროის აღქმისთვის. ამ feature engineering-ზე დაყრდნობით გავუშვი შემდეგი run-ები:

### ეტაპი 4: Single-Split Validation-ის სისუსტის აღმოჩენა → Time-Series Cross-Validation

Kaggle-ის რეალურმა score-მა (3111) მკვეთრად გადააჭარბა ჩვენს validation WMAE-ს (1927) — ეს შეუსაბამობა მიმანიშნებელი იყო, რომ single-split validation (`split_date='2012-01-01'`) რაღაცას ხელოვნურად ამარტივებდა.

მიზეზის დასადგენად გავუშვი **4-fold expanding-window time-series CV**, თითოეული fold ~3-თვიან validation window-ზე:

| Fold (validation პერიოდი) | Validation WMAE |
| :--- | :--- |
| 2011-10-01 → 2012-01-01 (Thanksgiving + Christmas) | **$3,267.95** |
| 2012-01-01 → 2012-04-01 | $2,095.98 |
| 2012-04-01 → 2012-07-01 (holiday-ის გარეშე) | $1,835.76 |
| 2012-07-01 → 2012-10-01 | $1,608.79 |
| **საშუალო (CV)** | **$2,202.12 ± $737.90** |

**აღმოჩენა:** ის ერთადერთი fold, რომელიც მოიცავს Thanksgiving/Christmas-ს, თითქმის **2x უფრო ცუდია** დანარჩენებზე. ორიგინალური single-split validation-ი სწორედ ამ ყველაზე რთულ პერიოდს გამორიცხავდა (ვინაიდან split-ი 2012-01-01-ზე იყო დაყენებული), რის გამოც validation WMAE ხელოვნურად დაბალი გამოდიოდა. **Kaggle-ის რეალური test-სეტი (2012 ნოემბერი – 2013 ივლისი) სწორედ ამ "რთული" holiday-სეზონის მსგავსია**, რის გამოც public score (3111) ბევრად უფრო ახლოსაა CV-ის holiday-fold-თან (3268), ვიდრე ორიგინალურ validation-თან (1927).

**დასკვნა:** Single time-split validation არასაკმარისია ამ დატასეტისთვის — საჭიროა multi-fold time-series CV რეალური განზოგადების შესაფასებლად.

### ეტაპი 5: დამატებითი ცდები CV-ის გამოყენებით

CV-ის დამკვიდრების შემდეგ, სამი დამატებითი ჰიპოთეზა შემოწმდა **იმავე 4 fold-ზე**, სამართლიანი შედარებისთვის:

| ექსპერიმენტი | მიდგომა | CV Mean WMAE |
| :--- | :--- | :--- |
| **Baseline (ლაგებით, ორიგინალი params)** | `Lag_51/52/53` + `Max_Sales_Yearly_Window`, `max_depth=5`, `n_estimators=600` | **$2,202.12 ± $737.90** ✅ |
| ახალი ჰიპერპარამეტრები | `max_depth=8`, `n_estimators=400`, `lr=0.05` | $2,218.80 ± $817.33 |
| ლაგების გარეშე | იგივე ახალი params, ლაგ-ფიჩერების გარეშე | $2,854.89 ± $562.76 ❌ |
| Holiday-ტიპის ფიჩერები | + `Is_Thanksgiving`, `Is_Christmas`, `Is_SuperBowl`, `Is_LaborDay`, `Weeks_To_Christmas` | $2,213.37 ± $753.66 |

**დასკვნები:**
- **ლაგების მოშორებამ WMAE ~30%-ით გააუარესა** — ეს ცხადყოფს, რომ `Lag_51/52/53` და `Max_Sales_Yearly_Window` ფიჩერები კრიტიკულია მოდელის სიზუსტისთვის.
- **ახალი (უფრო ღრმა) ჰიპერპარამეტრები არ სჯობს ორიგინალს** — `max_depth=8` სავარაუდოდ ოდნავ overfit-ს იწყებს ამ დატასეტზე `max_depth=5`-თან შედარებით.
- **ცალკეული holiday-ტიპის (Thanksgiving/Christmas/SuperBowl) dummy-ფიჩერების დამატებამ ვერ გააუმჯობესა შედეგი** — XGBoost-ის tree-based split-ები უკვე თავად სწავლობდა ანალოგიურ პატერნებს `IsHoliday` + `Month`/`Week`-ის კომბინაციიდან, ცალკეული dummy-ები კი ზედმეტი აღმოჩნდა მცირე მონაცემზე (~2,900 სტრიქონი თითო holiday-ზე).

საბოლოო არჩევანი: ორიგინალური მოდელი (`Lag_51/52/53` + `Max_Sales_Yearly_Window`, ორიგინალური ჰიპერპარამეტრები) რჩება საუკეთესოდ ყველა ტესტირებულ ვარიანტს შორის, დადასტურებული როგორც CV-ით (WMAE $2,202), ისე რეალური Kaggle submission-ით (public: 3111.21, private: 3223.04).

## LightGBM
LightGBM ახალი მოდელი იყო ჩემთვის და მუშაობის დაწყებამდე ზოგადად შევისწავლე მისი მუშაობის პრინციპი. არის ხის მოდელი რომელიც იყენებს გრადიენტ ბუსტინგს და ტრადიციული ალგორითმებისგან (მაგალითად XGBoost) განსხვავებით იყენებს Leaf-wise (Best-first) ხეების ზრდის სტრატეგიას. ნაცვლად იმისა, რომ ხე გაიზარდოს დონეების მიხედვით სიღრმეში, LightGBM ეძებს იმ კონკრეტულ ფოთოლს, რომელსაც აქვს ყველაზე დიდი დანაკარგი და მას ზრდის. ეს ტაქტიკა მოდელს აძლევს უნარს, ბევრად უფრო სწრაფად მიაღწიოს მაღალ სიზუსტეს. 

#
დროითი მწკრივების ამოცანებში მონაცემების სწორ მომზადებას გადამწყვეტი მნიშვნელობა აქვს. სამწუხაროდ ეს მწარედ გამოვცადე თავიდან, როცა data leakage მქონდა. დროის განგრძობითობა რომ არ დამეკარგა და მოდელს სწორად აღექვა მონაცემთა ქრონოლოგია, lag-ები დავამატე(sales_lag_52 - წინა წელი). პირველ ჯერზე ეს ლაგები არია და ქრონოლოგია მოირღვა. ამან განაპირობა საოცარი ფაბლიქ სქორი საბმიშენზე. შემდგომ მივუბრუნდი და სწორი პრეპროცესინგი ვცადე.
ლაგების გარდა ასევე სხვა მოდიფიკაციებიც გახდა საჭირო. WalmartDataTransformer კლასი ამ ყოველივეს პაიპლაინია. თავდაპირველად მოხდა Walmart-ის სამი ძირითადი წყაროს (train.csv/test.csv, features.csv და stores.csv) ინტეგრაცია Store და Date სვეტების მიხედვით. თარიღის სვეტი გადავიდა datetime ფორმატში, რათა მისგან შეგვძლებოდა დროითი კომპონენტების ამოღება. რადგანაც გაყიდვებს აქვს მკაფიო სეზონური ხასიათი (მაგალითად, ზაფხულის პერიოდი, სასკოლო სეზონის დაწყება, წლის ბოლო), თარიღიდან გამოვყავით წელი და კვირა. Walmart-ის გაყიდვებში ყველაზე დიდი პიკები ემთხვევა ოთხ ძირითად დღესასწაულს: Super Bowl, Labor Day, Thanksgiving და Christmas. იმის გამო, რომ სადღესასწაულო ციებ-ცხელება იწყება დღესასწაულამდე ერთი კვირით ადრე და გრძელდება მომდევნო კვირაშიც, ჩვენ შევქმენით წინა და შემდგომი კვირის ფიჩერები (Lag Features) გლობალურ დონეზე (groupby(['Store', 'Dept']) ჭრილში):
Is_Pre_Holiday: აღნიშნავს, არის თუ არა მომავალი კვირა უქმე დღე (shift(-1)).
Is_Post_Holiday: აღნიშნავს, იყო თუ არა წინა კვირა უქმე დღე (shift(1)).
ეს ნაბიჯი კრიტიკული აღმოჩნდა ვალიდაციისა და ტესტირების ფაზაში, რადგან მოდელს მისცა მეხსიერება, თუ რა ხდება დღესასწაულების გარშემო. LightGBM-ს აქვს უნიკალური და ძალიან ეფექტური შიდა მექანიზმი კატეგორიული ცვლადების (მაგალითად, მაღაზიის ტიპები, მაღაზიის ნომრები და დეპარტამენტები) დასამუშავებლად. One-Hot Encoding-ის ნაცვლად, რომელსაც შეუძლია მატრიცის ზედმეტად გაფართოება, ყველა ობიექტის ტიპის სვეტი გარდაიქმნა explicit category ტიპად, რაც მოდელს საშუალებას აძლევს ოპტიმალურად დაყოს მონაცემები კატეგორიების მიხედვით.

#
უშუალოდ მოდელის წვრთნა საკმაოდ roller-coaster ride პროცესი იყო.
ძირითადად სამი ტიპის ჰიპერპარამეტრს ვთრექავდი. ვცვლიდი ხეების რაოდენობას (n_estimators), მათ სიღრმესა და ფოთლების რაოდენობას (num_leaves). მეორე კომპონენტი იყო სწავლის ტემპი (learning_rate) ბუსტინგისთვის, რომელსაც ხან ავუწევდი და ხან დავუწევდი, რომ მენახა, რამდენად სწრაფად ან ფრთხილად ახდენდა ალგორითმი ოპტიმიზაციას. და მესამე, რა თქმა უნდა, რეგულარიზაციის პარამეტრები (L1/L2 ჯარიმები ხეებისთვის), რაც დამეხმარა ოვერფიტინგის კონტროლში.

მთელი ამ ექსპერიმენტების დროს, MLflow-ს მეშვეობით აბსოლუტურად ყველა გაშვებაზე ზუსტად ერთსა და იმავე მეტრიკებსა და პარამეტრებს ვთრექავდი, რათა მოდელების შედარება სამართლიანი ყოფილიყო. პირველ რიგში, ვინახავდი მოდელის ყველა კონფიგურაციას, რომ ზუსტად გვცოდნოდა, რომელმა პარამეტრმა რა შედეგი მოიტანა. მეტრიკებიდან კი მთავარი აქცენტი გადატანილი იყო სატრენინგო და სავალიდაციო WMAE-ზე , სადაც სადღესასწაულო კვირებს 5-ჯერ მეტი წონა ჰქონდა. ამ ორ ციფრს შორის სხვაობაზე დაკვირვებით მაშინვე ვხვდებოდით, მოდელი რეალურად სწავლობდა თუ უბრალოდ იზეპირებდა დატას. გარდა ამისა, ვთრექავდით მოდელის სქემას (Signature) და თითოეული ექსპერიმენტის ვერსიებს Model Registry-ში, რამაც საშუალება მოგვცა, ფინალურ ეტაპზე მთელი ეს ისტორია გადაგვეხედა, ცივი გონებით შეგვედარებინა ციფრები და მათ შორის ყველაზე დაბალი სავალიდაციო შეცდომის მქონე მოდელი გამოგვეცხადებინა ჩვენს საბოლოო ჩემპიონად.


მანამ, სანამ რთულ კონფიგურაციებზე გადავიდოდი, საწყისი წერტილისთვის ავიღე ბეისლაინ მოდელი სახელით baseline_lightGBM. აქ ალგორითმს საერთოდ არ შევეხე და LGBMRegressor გავუშვი აბსოლუტურად სუფთად, თავის სტანდარტულ 100 ხეზე და 31 ფოთოლზე, ყოველგვარი დამატებითი ლაგების გარეშე.
შეფასება აქაც მკაცრად WMAE-ით გავაკეთე, სადაც სადღესასწაულო კვირებს 5-იანი წონა მივაბი. ამ ბეისლაინმა დამანახა, რომ მოდელი ტრენდს კი ხვდებოდა, მაგრამ ქულა საკმაოდ მაღალი დაჯდა, რადგან 100 ცალი პატარა ხე ფიზიკურად ვერ ტევდა Walmart-ის მთელ სირთულეს და დღესასწაულების გიგანტურ პიკებს აზიანებდა. თუმცა, ზუსტად ამან მომცა მიმართულება, რომ მოდელს მეტი სიმძლავრე და ისტორიული ფიჩერები სჭირდებოდა.

### მთავარი მიგნებები ექსპერიმენტების მსვლელობისას:
* **გარე რეგრესორებმა (External regressors) მოდელი დააზიანა და არ გააუმჯობესა.** ~3,254 დამოუკიდებელი სერიისა და ხშირად თითო სერიაზე ~150-ზე ნაკლები კვირეული დაკვირვების პირობებში, 4–5 რეგრესორის დამატება Prophet-ს ზედმეტ თავისუფლებას აძლევს. მოდელი რეალური სიგნალის ნაცვლად ხმაურს (noise) სწავლობს ტემპერატურის, CPI-ისა და უმუშევრობის მონაცემებში. რეგრესორების ორივე ვარიანტმა (ექსპ. 2, 3) Plain Prophet-ზე ცუდი ქულა აჩვენა.
* MarkDown მონაცემები არ ყოფილა კონკრეტული დამნაშავე — მხოლოდ მისი ამოღებით ბეზლაინის ქულა ვერ აღდგა, ასე რომ, ფართო დღესასწაულების ფანჯარა (ექსპ. 2 vs 3) ან ზოგადად რეგრესორების ჯგუფი უფრო სავარაუდო მიზეზად ჩავთვალე
* **დაბალი `changepoint_prior_scale` (უფრო რეგულარიზებული ტრენდი)** სტაბილურად ამცირებს როგორც ვალიდაციის შეცდომას, ასევე სხვაობას train-სა და val-ს შორის. ეს არის ოვერფიტინგის ყველაზე სუფთა და გასაგები გამოსწორება, რაც კი ვიპოვე, და თან იმპლემენტაციის სირთულის მხრივ არაფერი დაგვიჯდა (რეგრესორებისგან განსხვავებით).
* **მულტიპლიკაციურმა სეზონურობამ (`multiplicative seasonality`) არ უშველა.** მიუხედავად თეორიული საფუძვლისა (შობის ეფექტი მასშტაბში უნდა იზრდებოდეს მაღაზიის ზომასთან ერთად), მან ოდნავ უარესი ქულა აჩვენა, ვიდრე ადიტიურმა (5037.35 vs 5003.30) იმავე `cps=0.01` პარამეტრზე.
* **`holidays_prior_scale` არ არის მნიშვნელოვანი ბერკეტი ამ ქეისში.** ტესტირებამ 300-სერიიან ქვესიმრავლეზე 0.1-დან 30.0-მდე (ნაგულისხმევი არის 10.0) აჩვენა, რომ მხოლოდ აგრესიულად დაბალმა მნიშვნელობამ (0.1, რომელიც სრულად ახშობს დღესასწაულების ეფექტს) დააზიანა მოდელი. 1.0-დან 30.0-მდე შეცდომა თითქმის უცვლელი დარჩა (~0.06% სხვაობა, ანუ ხმაური). შესაბამისად, სრულ სეტზე გაშვება არ ღირდა და დარჩა Prophet-ის default (10.0).
* სხვაობა შეცდომაში შემყვან მარტივ ფანჯარასა (1768.75) და რეალურ რთულ ფანჯარას (~5000) შორის არის შეხსენება, რომ ვალიდაცია ყოველთვის ყველაზე რთულ/რეპრეზენტაციულ პერიოდზე უნდა გავაკეთოთ და არა უბრალოდ "ბოლო N კვირაზე".

##  პაიპლაინად შეფუთვა (ინფერენსისთვის)
XGBoost ბეზლაინისგან განსხვავებით (რომელიც ერთი sklearn Pipeline ობიექტია), ეს პროექტი აშენებს ~3,254 დამოუკიდებელ Prophet მოდელს — თითოეულს (Store, Dept) წყვილისთვის. ეს მოდელები მოქცეულია `ProphetPerSeriesForecaster` კლასში (`prophet_pipeline.py`), რომელსაც აქვს `.fit(train_df)` / `.predict(future_df)` მეთოდები, რათა გარედან ისე გამოიყურებოდეს, როგორც ერთიანი მოდელი.

# Deep Learning 

## NBEATS
N-BEATS შედგება ერთმანეთზე გადაბმული ბლოკებისგან (Blocks), რომლებიც ერთიანდებიან სეგმენტებად (Stacks). მოდელი მუშაობს Residual (ნარჩენი) კავშირების პრინციპით ორმხრივად.

Backcast ($\boldsymbol{x}$): თითოეული ბლოკი ცდილობს აღადგინოს ისტორიული მონაცემები (უკანა ხედვა) და დაინახოს, რა შეცდომა დაუშვა. ეს შეცდომა აკლდება საწყის სიგნალს და შემდეგი ბლოკი მხოლოდ "დარჩენილ" ხმაურს სწავლობს.

Forecast ($\boldsymbol{\hat{y}}$): თითოეული ბლოკი პარალელურად აკეთებს მომავლის პროგნოზს. საბოლოო პროგნოზი არის ყველა ბლოკის მიერ დაგენერირებული შედეგების მარტივი ჯამი.

**დასკვნა**: ცხრა+ სხვადასხვა მიმართულების ცდის შემდეგ (steps grid, structural decomposition, longer input, weight decay, dropout, early stopping), **`nbeats_baseline` (200 steps, default config) დარჩა უცვლელად საუკეთესო** — val_wmae=1344.48.

### Baseline

პირველი, მინიმალური კონფიგურაცია (`input_size=52`, `max_steps=200`, exogenous features გარეშე):

### Sanity check 

Baseline-ის საეჭვოდ დაბალმა val_wmae-მ (1,499) ეჭვი გააჩინა leakage-ზე. ამის შესამოწმებლად დავამატეთ ორი ტრივიალური baseline იმავე val split-ზე:

| Baseline | val_wmae |
|---|---|
| Naive (last value carried forward) | 2,415.07 |
| Naive (seasonal, t-52 კვირა) | 1,718.32 |
| **PatchTST baseline** | **1,499.02** |

**დასკვნა:** PatchTST ორივე ტრივიალურ baseline-ს სჯობდა — ეს ნიშნავს, რომ 1,499 რეალური, არა-leaked შედეგია. ასევე `train_wmae ≈ val_wmae` სიახლოვე ჯანსაღი, არა-overfit-ული სურათია.

### აღმოჩენილი შეზღუდვა — PatchTST-ს არ შეუძლია exogenous ცვლადები

თავდაპირველი გეგმა იყო markdown/holiday/temperature/CPI-ის ჩამატება `futr_exog_list`/`hist_exog_list`-ის საშუალებით. მაგრამ ვნახე და  Runtime-მაც დაადასტურა, რომ **PatchTST-ს (ამ ბიბლიოთეკაში) საერთოდ არ შეუძლია exogenous ცვლადების მიღება** — არც future, არც historical, არც static. ეს არქიტექტურული შეზღუდვაა.

ფოკუსი გადავიტანე PatchTST-ის საკუთარ hyperparameter-ებზე (`input_size`, `patch_len`, `stride`, `max_steps`), ნაცვლად exogenous features-ისა. 

### Hyperparameter sweep

ორ რაუნდად ჩატარდა, ყოველი ცდა MLflow-ში ცალკე run-ად დალოგიი:

| Run | input_size | patch_len / stride | max_steps | val_wmae |
|---|---|---|---|---|
| baseline | 52 | 16 / 8 | 200 | 1,499.02 |
| steps=300 | 52 | 16 / 8 | 300 | 1,546.68 |
| steps=600 | 52 | 16 / 8 | 600 | 1,442.01 |
| longer_input | 104 | 16 / 8 | 600 | 1,417.38 |
| finer_patches | 104 | 8 / 4 | 600 | 1,402.16 |
| even_finer | 104 | 4 / 2 | 600 | 1,380.01 |
| **finer_more_steps** | **104** | **8 / 4** | **1000** | **1,362.83** ✅ |


**ტენდენციები:**
- `input_size` 52→104 (2 წლიანი ისტორია, ორი სრული სეზონური ციკლი) — მკაფიო გაუმჯობესება
- უფრო წვრილი `patch_len`/`stride` (16/8 → 8/4) — ასევე ეხმარება
- `patch_len=4`-მდე შემცირება (`even_finer`) — **უარესდება** 8-თან შედარებით, ანუ 8 ახლოსაა ოპტიმალურთან ამ დატასეტისთვის
- `max_steps` 600→1000-მდე გაზრდა საუკეთესო კონფიგურაციაზე — კიდევ სჯობა

**Lightning-ის internal early stopping (`early_stop_patience_steps`)** რამდენჯერმე ჩავარდა `ptl/val_loss` metric-key mismatch-ის გამო. ამიტომ გადავწყვიტეთ გამოგვეყენებინა fixed `max_steps` სხვადასხვა კონფიგურაციებში + ჩვენი საკუთარი `val_wmae` შედარება იძლევა იმავე ინფორმაციას (სად იწყება overfitting).

## D-Linear

## მოდელის შესახებ

DLinear ჩემთვის იყო ერთ-ერთი ყველაზე მოულოდნელი აღმოჩენა ამ პროექტში. სანამ დავიწყებდი, ჩავიკითხე Zeng et al. (2022)-ის ნაშრომი "Are Transformers Effective for Time Series Forecasting?", სადაც ავტორები აჩვენებენ, რომ მძიმე, transformer-ზე დაფუძნებული არქიტექტურები (Informer, Autoformer, FEDformer) ხშირად ვერ ჯობნიან ერთ ჩვეულებრივ ხაზოვან მოდელს. ეს თეზისი მაინტერესებდა Walmart-ის დატაზე როგორ იმუშავებდა.

არქიტექტურულად DLinear მარტივია. სერია იშლება ორ ნაწილად:
- **Trend** — moving average-ით ამოღებული გლუვი კომპონენტი,
- **Seasonal (residual)** — ის, რაც trend-ის გამოკლების შემდეგ რჩება.

თითოეულ ამ ნაწილს ცალკე ერთი linear layer ასწავლის, საბოლოო პროგნოზი კი ორივეს ჯამია. ამის გამო მოდელს არ აქვს `hidden_size` ან `depth` პარამეტრი — capacity-ს ერთადერთი წყარო training steps-ის რაოდენობაა.

გამოვიყენე `neuralforecast` ბიბლიოთეკის იმპლემენტაცია, ტესტირება და ლოგირება კი  MLflow + DagsHub-ის მეშვეობით, ისევე როგორც LightGBM-ის ექსპერიმენტებში.

#

DLinear-ს (და ზოგადად `neuralforecast`-ს) სჭირდება long-format დატა სამი სვეტით: `unique_id`, `ds`, `y`. თითოეული (Store, Dept) წყვილი აქ ცალკე დამოუკიდებელ time series-ად განიხილება, ანუ 45 მაღაზია × 81 დეპარტამენტი თეორიულად 3331 ცალკე მწკრივს იძლევა.

პრობლემა ის იყო, რომ ბევრი ეს კომბინაცია რეალურად თითქმის ცარიელი იყო, ზოგს ერთი-ორი observation თუ ჰქონდა. ამიტომ **Run 1 (`DLinear_Preprocessing`)**-ში დავაწესე ბარიერი: სერია უნდა შეიცავდეს მინიმუმ 80 observation-ს (`MIN_SERIES_LENGTH = 80`), თორემ მოდელს არც input window-სთვის (52 კვირა) და არც ვალიდაციისთვის საკმარისი მონაცემი არ ეყოფოდა. ამ ფილტრმა 3331 სერიიდან 2934 დატოვა, ჯამში 412,935 observation.

Validation split-იც chronological ლოგიკით გავაკეთე. თითოეული სერიის ბოლო 12 კვირა (`FORECAST_HORIZON = 12`) გამოვყავი validation-ისთვის, დანარჩენი დარჩა training-ისთვის. ეს მნიშვნელოვანია, რადგან random split აქ დროის განგრძობითობას დაარღვევდა და leakage-ს გამოიწვევდა, ზუსტად ის შეცდომა, რაც LightGBM-ის ეტაპზეც თავიდან დამემართა lag-ების არასწორად დამატებისას.

 Run 3 `DLinear_ExtendedKernel`

პირველი ჰიპოთეზა, რომელიც შევამოწმე, ეხებოდა `moving_avg_window`-ს (kernel size), რომელიც განსაზღვრავს, რამდენად "გლუვია" trend კომპონენტი:

- **25** ≈ ნახევარწლიანი smoothing (short-term trend),
- **51** ≈ სრულწლიანი smoothing — თითქმის მთელი წლიური ციკლი trend-იდან იხსნება.

იდეა იყო, რომ თუ trend სრულ წლიურ ციკლს "შთანთქავს", დარჩენილ seasonal residual-ში Walmart-ის მკვეთრი holiday spikes უფრო სუფთად და გამოკვეთილად დარჩება, რაც მოდელს დღესასწაულების დაჭერაში დაეხმარება.

დანარჩენი ჰიპერპარამეტრები ბეისლაინის იდენტური დავტოვე, მხოლოდ `moving_avg_window: 25 → 51`.

**შედეგი WMAE: 1520.39**

მცირე, მაგრამ სტაბილური გაუმჯობესება — ჰიპოთეზა დადასტურდა.

 Run 4  `DLinear_ExtendedTraining`

რადგან DLinear-ს, როგორც ვთქვი, არ აქვს capacity-related ცვლადები (layers, hidden units), ერთადერთი ბერკეტი, რაც დარჩა, training-ის ხანგრძლივობა და სიფრთხილეა. ამიტომ ამ გაშვებაში:

- `max_steps: 500 → 1500` (3x მეტი optimization step),
- `learning_rate: 1e-3 → 5e-4` (უფრო ნაბიჯი-ნაბიჯი, "ფრთხილი" convergence),
- `moving_avg_window` დავტოვე 51-ზე (Run 3-ის შედეგი).

იდეა მარტივია: მცირე learning rate-ს მეტი დრო სჭირდება ოპტიმუმამდე მისასვლელად, ამიტომ steps-ის რაოდენობაც ავწიე პროპორციულად, რომ ტრენინგს არ გამოვაცილო ნაადრევად.

**შედეგი WMAE: 1497.78**  (საუკეთესო შედეგი სამივე გაშვებიდან)

სამივე run-ის შედარებამ (scoreboard-ის მეშვეობით) ცხადყო, რომ `extended_training` კონფიგურაცია ყველაზე დაბალ ვალიდაციურ WMAE-ს იძლევა:

# Arima and Sarima
ARIMA და SARIMA ორივე კლასიკური სტატისტიკური მოდელებია დროითი მწკრივების პროგნოზირებისთვის, LightGBM-ისგან და სხვა ხის მოდელებისგან სრულიად განსხვავებული ლოგიკით. ARIMA (AutoRegressive Integrated Moving Average) სამი კომპონენტისგან შედგება: AR (Autoregressive) ნაწილი პროგნოზს აგებს წინა პერიოდების საკუთარ მნიშვნელობებზე დაყრდნობით, I (Integrated) არის დიფერენცირების ხარისხი, რომელიც მონაცემს არასტაციონარულიდან სტაციონარულ სახეს აძლევს (ანუ ტრენდს აშორებს), ხოლო MA (Moving Average) წინა პროგნოზების შეცდომებზე დაყრდნობით ასწორებს შემდეგ პროგნოზს. SARIMA ამ სამივეს ამატებს სეზონურ განზომილებას - იგივე სამი კომპონენტი (P, D, Q), მაგრამ გამეორებული სეზონურ ლაგზე (ჩვენს შემთხვევაში 52 კვირა, ანუ წელიწადი), რაც თეორიულად საშუალებას უნდა მიეცა მოდელისთვის დღესასწაულების და წლიური ციკლების დაჭერა, რასაც უბრალო ARIMA ვერ აკეთებს.



რადგან ARIMA/SARIMA უნივარიატული (ერთგანზომილებიანი) მოდელებია და არ იღებენ Store/Dept დონის ფიჩერებს ისე, როგორც LightGBM, პირველი ნაბიჯი მონაცემის აგრეგირება იყო. ყველა მაღაზიისა და დეპარტამენტის კვირეული გაყიდვები ავჯამე ერთ საერთო კვირეულ მწკრივად (national level weekly sales). თარიღი გადავიყვანე datetime ფორმატში და დავაფიქსირე კვირეული სიხშირე (W-FRI, ვინაიდან Walmart-ის მონაცემები პარასკევებზეა აწყობილი). Feature engineering-ის მაგივრად, რაც ხის მოდელებში გავაკეთე, აქ მთელი "ინტელექტი" თავად მოდელის (p,d,q)(P,D,Q,s) პარამეტრებშია დამალული.
ვალიდაციისთვის ბოლო 12 კვირა გამოვყავი Hold-out ნაკრებად, დანარჩენი კი დატრენინგებისთვის გამოვიყენე. საზომ მეტრიკად აქაც იგივე WMAE დავტოვე, სადაც სადღესასწაულო კვირებს 5-ჯერ მეტი წონა აქვს - ეს იმისთვის გავაკეთე, რომ ARIMA/SARIMA-ს შედეგები LightGBM-ის შედეგებთან პირდაპირ შედარებადი ყოფილიყო. თითოეული გაშვება MLflow-ში ვთრექავდი (order, seasonal_order, train/val WMAE, AIC, BIC პარამეტრები და პროგნოზის გრაფიკი, ასევე თავად fitted მოდელი), ხოლო remote tracking-ისთვის DagsHub გამოვიყენე.
გავუშვი რამდენიმე კონფიგურაცია, ორი წმინდა ARIMA (arima_1_1_1, arima_2_1_2 - სეზონურობის გარეშე) და ორი SARIMA (sarima_1_1_1_52, sarima_2_1_1_52 - 52-კვირიანი სეზონურობით), რომ დამენახა, სეზონური კომპონენტის დამატება საერთოდ არის თუ არა გამართლებული ამ დატაზე.


შედეგი, რბილად რომ ვთქვა, იმედგამაცრუებელი იყო. ველოდი რომ თაიმსერიებზე მორგებული მოდელები უკეთეს შედეგებს დადებდნენ. WMAE ორივე მოდელისთვის (ARIMA-ც და SARIMA-ც)  რამდენიმე ათეულჯერ მეტი გამოვიდა, ვიდრე LightGBM-ის საუკეთესო გაშვებამ (LGBM_MAE_Leaves_63, WMAE ≈ 1809.84) აჩვენა. სეზონურობის დამატებამ (SARIMA) გარკვეულწილად გააუმჯობესა სუფთა ARIMA-სთან შედარებით, მაგრამ მაინც ვერანაირად ვერ მიუახლოვდა ხის მოდელების სიზუსტეს.

ჩემი აზრით ამის რამდენიმე მიზეზი არსებობდა. პირველ რიგში, აგრეგირებამ ერთ საერთო კვირეულ მწკრივში სრულად წაშალა Store და Dept დონის ინფორმაცია. სწორედ ეს გრანულარობა იყო ის, რაც LightGBM-ს დღესასწაულების ირგვლივ სპაიკების ზუსტად დაჭერაში ეხმარებოდა (Is_Pre_Holiday/Is_Post_Holiday ფიჩერების მეშვეობით). ARIMA/SARIMA-ს ამის გაკეთება არ შეუძლია, რადგან ცალკეული (p,d,q) პარამეტრებით მთელი წლის რთული, არაწრფივი და ხმაურიანი პატერნი უნდა აღეწერა, რაც პრაქტიკულად შეუძლებელია ასეთი მარტივი ხაზოვანი მოდელით. მეორეც, სადღესასწაულო პიკები თავისი ბუნებით მკვეთრი და არაწრფივია, SARIMA-ს სეზონური კომპონენტი კი უფრო გლუვ, განმეორებად ციკლებზეა მორგებული და არა მკვეთრ, მოკლევადიან შოკებზე. შედეგად, პროგნოზი მუდმივად "აჭარბებდა" წყნარ პერიოდებს და ვერ იჭერდა დღესასწაულების უეცარ აწევებს, რაც WMAE-ში (სადაც სწორედ დღესასწაულებს 5x წონა აქვს) განსაკუთრებით მძიმედ აისახა.

ეს ექსპერიმენტი მაინც ღირებული იყო იმ კუთხხით, რომ დამანახა, რომ ამ ტიპის მონაცემისთვის (მრავალი store/dept, ძლიერი კატეგორიული ეფექტები, მკვეთრი დღესასწაულო არაწრფივობა) კლასიკური სტატისტიკური მოდელები ხის მოდელებთან შედარებით აშკარად არასაკმარისია, და დავრწმუნდი რომ ხის მოდელებზე ფოკუსირება სწორი გადაწყვეტილება იყო.
